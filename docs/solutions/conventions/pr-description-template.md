# PR 설명은 한국어 4섹션 템플릿(문제·변경·테스트·데모)을 따른다

## Context

`sreworkspaces`의 MR 설명 컨벤션은 문제·변경·테스트·데모 4섹션으로 리뷰 맥락을 정리한다. `zWorkspaces`에서도 PR/MR 본문은 같은 구조를 기본으로 사용한다. 사용자가 영어로 요청하지 않는 한, PR 제목과 본문은 한국어로 작성한다.

## Guidance

PR/MR 본문은 아래 4섹션을 기본 골격으로 사용한다. 실제 PR 본문에는 안내 문구나 예시용 문장을 남기지 말고, 각 섹션을 실제 내용으로 채운다.

```markdown
## 해결하려는 문제가 무엇인가요?
<배경·상황만 작성한다. 해결 방법은 여기서 설명하지 않는다. 관련 이슈·문서·상위 PR이 있으면 링크한다.>

## 변경 사항
<필수. 이 PR에서 바뀐 내용을 리뷰 가능한 단위로 간단히 나열한다.>

## 테스트 체크리스트
<필수. 실제 수행한 검증을 체크리스트로 쓴다. 실행하지 못한 항목은 미실행 사유를 명시한다.>

## 데모
<선택. 확인 방법, 명령, 화면, PR 의존성 등을 적는다. 없으면 "해당 없음"으로 둔다.>
```

작성 규칙:

- **한국어 기본값:** 사용자가 영어를 명시하지 않으면 PR 제목과 본문을 한국어로 작성한다.
- **문제와 해결 분리:** 문제 섹션에는 배경·불편·리스크만 쓰고, 해결책은 변경 사항에 쓴다.
- **변경 사항·테스트 체크리스트 필수:** 검증은 `- [x]` 체크리스트로 작성한다.
- **불확실성 표시:** CI가 없거나 외부 인증/서비스가 없어 검증하지 못한 항목은 숨기지 말고 미실행 사유를 쓴다.
- **민감 정보 금지:** API key, token, 내부 호스트, 개인 경로는 노출하지 않는다. 필요하면 `<로컬 경로>`, `<프록시 URL>`처럼 일반화한다.
- **자동 생성 노이즈 제거:** `Generated by`, 변경 파일 자동 목록, 템플릿 안내용 `>` blockquote는 남기지 않는다.

## Why This Matters

- 리뷰어가 문제 → 변경 → 검증 순서로 PR을 빠르게 읽을 수 있다.
- 한국어 작업 맥락을 유지해, 사용자가 별도로 번역하거나 PR 본문을 다시 쓰지 않아도 된다.
- 자동화 에이전트가 PR을 만들 때도 일관된 품질의 설명을 남긴다.

## When to Apply

- `ce-commit-push-pr`, `/lfg`, 수동 `gh pr create/edit` 등 모든 PR 생성·갱신 흐름.
- GitLab MR에도 같은 4섹션을 적용한다. 이름만 MR일 뿐 본문 구조는 동일하다.
- 코드 변경, 문서 변경, 계획/스펙 변경 모두에 적용한다. 다만 데모가 없으면 `해당 없음`으로 둔다.

## Example

```markdown
## 해결하려는 문제가 무엇인가요?
`oh-my-pi`를 로컬 Pi 배포판처럼 키우는 작업이 여러 PR로 나뉘면서, 각 PR이 어떤 배경에서 나온 변경인지 리뷰어가 한눈에 보기 어렵다.

## 변경 사항
- Capability Registry를 추가해 extension/capability 메타데이터를 한 곳에서 관리한다.
- env-loader의 비활성화 알림이 registry의 toggle 정보를 사용하도록 바꾼다.
- 구현 계획 문서를 추가해 ideation 문서와 실제 변경 사이의 연결을 남긴다.

## 테스트 체크리스트
- [x] `npm ci`
- [x] `npm exec tsc -- --noEmit --module NodeNext --moduleResolution NodeNext --target ES2022 --strict --skipLibCheck ...`
- [x] `git diff --check`

## 데모
해당 없음. 런타임 동작은 Pi 세션 시작 시 env-loader 알림으로 확인한다.
```
