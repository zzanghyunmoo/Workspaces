## 제품 요약

Our Moneyflow는 개인이 여러 은행 계좌를 읽기 전용으로 연결하고, 각 계좌를 하나의
예산 봉투처럼 운영하도록 돕는 Flutter 기반 Android/iOS 모바일 앱이다. 사용자는 은행
잔액과 실제로 더 써도 되는 예산 잔액을 구분하고, 이번 주·이번 달의 안전 사용 가능액과
지난달 동일 경과기간 대비 지출을 확인하며, 주·월·연 단위 보고서를 볼 수 있다.

1차 제품은 한국어·원화·개인 명의 입출금 계좌를 대상으로 한다. 국민은행,
카카오뱅크, 토스뱅크의 공식 운영 연결이 모두 검증되고 Apple App Store와 Google
Play에 공개되어야 출시로 인정한다. 샘플·CSV·개발용 connector는 개발과 내부 검증에
사용하지만 실제 연결처럼 표시하지 않는다.

## 제품 압력 테스트

### 사용자의 실제 문제

- 여러 계좌에 흩어진 잔액만으로는 지금 얼마를 써도 안전한지 판단하기 어렵다.
- 은행 앱의 거래 목록은 예산, 분류 교정, 기간 비교와 결산이라는 생활 의사결정으로
  연결되지 않는다.
- 계좌 간 이체, 카드대금, 환불과 취소가 섞이면 단순 합산 보고서는 실제 소비를
  왜곡한다.
- 한 은행 연결이 지연되거나 만료되어도 총액이 완전한 것처럼 보이면 금융 앱에 대한
  신뢰가 무너진다.

### 제품이 지킬 약속

- 돈을 움직이지 않는다. 조회, 분류, 메모, 예산과 보고만 제공한다.
- 은행 잔액과 예산상 사용 가능액을 분리한다.
- 데이터 누락과 마지막 동기화 시각을 숨기지 않는다.
- 은행 원본은 덮어쓰지 않고 사용자 교정은 되돌릴 수 있게 기록한다.
- 샘플, 개발, 심사, 운영 연결 상태를 명시적으로 구분한다.

### 출시를 막는 조건

- 국민은행, 카카오뱅크, 토스뱅크 중 하나라도 공식 운영 읽기 연결이 없다.
- 송금·결제 등 쓰기 capability가 제품 또는 connector 계약에 포함된다.
- 계좌 간 이체, 환불·취소, 중복 거래를 다루는 정합성 검증이 실패한다.
- 일부 은행 데이터가 누락됐는데 전체 합계가 완전한 값처럼 표시된다.
- 계정 삭제, 연결 철회, 데이터 보존·삭제 경계 또는 스토어 개인정보 고지가 검증되지
  않았다.
- 고위험 보안 결함이나 금융 데이터가 로그·분석 도구로 유출되는 경로가 남아 있다.

## 사용자와 시스템 행위자

- 사용자: 한국의 단일 개인 사용자. 본인 명의 입출금 계좌를 연결하고 예산과 보고서를
  관리한다.
- 금융 연결 제공자: 금융결제원 오픈뱅킹 또는 운영 승인을 받은 공식 제공 경로.
  사용자 인증·동의·계좌 연결·조회·토큰 수명주기를 제공한다.
- 개별 은행: 국민은행, 카카오뱅크, 토스뱅크. 은행별 조회 지연과 장애가 독립적으로
  발생할 수 있다.
- 운영자: 연결 상태와 장애를 운영하되 사용자의 금융 원문을 불필요하게 열람할 수 없다.
- 앱 스토어 심사자: 개인정보, 금융 기능, 계정 삭제, 데이터 안전성과 앱 동작을
  검토한다.

## 제품 범위

### 포함

- Flutter로 만든 Android/iOS 앱
- 한국어, 원화, 단일 개인 사용자
- 국민은행·카카오뱅크·토스뱅크 개인 입출금 계좌의 공식 읽기 전용 연결
- 계좌 목록, 잔액, 거래 동기화와 연결 상태
- 계좌별 가계부: 분류, 메모, 사용자 교정과 되돌리기
- 계좌별 월 예산과 기간별 예산 해석
- safe-to-spend, 예산 소진률, 지난달 동일 경과기간 대비 지출
- 주·월·연 보고서
- 연결 만료·지연·장애·재인증·해제 흐름
- 개인정보·동의·보존·삭제·계정 삭제와 스토어 공개 요건
- 금융 원문을 담지 않는 운영 관측과 장애 대응

### 제외

- 송금, 자동이체, 결제, 계좌 개설 등 모든 쓰기 기능
- 공동 가계부, 가족·팀 공유, 다중 사용자 예산
- 신용카드, 대출, 투자, 보험, 가상자산
- 다중 통화와 해외 은행
- 웹 앱과 데스크톱 앱
- AI 자동 분류, 미래 지출 예측, 투자·신용 조언
- 오픈뱅킹·마이데이터 자격을 우회하는 스크래핑
- 1차 출시 이후의 알림·챌린지·소셜 기능

## 기능 요구사항

### R1. 공개 모바일 제품

- Android와 iOS에서 동일한 제품 계약을 제공한다.
- 한국 Google Play와 Apple App Store의 일반 사용자가 설치 가능한 공개 상태를 1차
  출시 완료로 본다.
- 내부 배포, TestFlight, 비공개 테스트, 샘플 데모는 출시 전 단계다.

### R2. 세 은행의 공식 읽기 전용 연결

- 사용자는 국민은행, 카카오뱅크, 토스뱅크의 본인 계좌를 공식 인증·동의 흐름으로
  연결할 수 있다.
- connector의 capability는 계좌 목록·잔액·거래 조회로 제한한다.
- 송금·결제 API와 권한은 요청하거나 저장하지 않는다.
- 공식 운영 승인과 계약이 완료되지 않은 연결은 운영 연결로 표시하지 않는다.

### R3. 동의와 연결 수명주기

- 연결 전 조회 범위, 보존 목적과 기간, 철회 방법을 설명한다.
- 은행별 상태를 연결 중, 정상, 지연, 만료, 장애, 재인증 필요, 해제로 구분한다.
- 사용자는 은행별 재인증과 연결 해제를 수행할 수 있다.
- 연결 해제는 토큰을 폐기하고 미래 동기화를 중단하되 과거 가계부·보고 이력은 유지한다.
- 사용자는 별도의 은행 데이터 삭제로 해당 연결의 원본·파생·캐시 데이터를 제거할 수 있다.

### R4. 변경 불가능한 은행 원본

- 수신한 원본 거래와 조회 메타데이터는 사용자 교정으로 덮어쓰지 않는다.
- 재동기화와 중복 수신에도 같은 원본 거래를 중복 집계하지 않는다.
- 금액, 통화, 발생·처리 시각, 계좌, 거래 식별과 수집 출처를 추적할 수 있다.

### R5. 정규화된 경제적 사건

- 원본 거래를 보고서에 사용하는 경제적 사건으로 정규화한다.
- 자기 계좌 간 이체는 수입·지출에서 제외하고 양쪽 거래를 연결할 수 있다.
- 신용카드 transaction feed는 연결하지 않는다. 카드대금으로 보이는 계좌 출금은 일반
  지출로 유지하고 사용자가 직접 제외·복원할 수 있으며 자동 중복 제거하지 않는다.
- 환불·취소·정정은 원거래와 연결해 순지출을 교정한다.
- 거래처 표기를 정규화하되 원문을 보존한다.

### R6. 계좌별 가계부와 가역적 교정

- 사용자는 계좌별 거래에 카테고리와 메모를 기록할 수 있다.
- 자동 또는 기본 분류가 있더라도 사용자의 수정 이력을 보존한다.
- 사용자 교정은 되돌릴 수 있고 은행 원본과 구분된다.
- 분류되지 않은 거래와 검토가 필요한 거래를 찾을 수 있다.

### R7. 계좌별 예산 봉투

- 사용자는 각 계좌에 월 예산을 설정·수정할 수 있다.
- 은행 잔액, 해당 기간의 예산, 확정 지출, 남은 예산을 각각 구분해 제공한다.
- 예산을 바꿔도 과거 원장과 은행 잔액을 변경하지 않는다.
- 예산 합계에 포함된 계좌와 제외된 계좌를 알 수 있다.

### R8. 행동 지표

- 홈은 `남은 예산 = 월 예산 - 확정 순지출`과
  `오늘부터 일평균 사용 가능액 = max(남은 예산, 0) / 오늘 포함 남은 일수`를 분리한다.
- 월 예산 소진률과 현재 경과율을 함께 비교한다.
- 지난달 비교는 달력 전체가 아니라 동일 경과기간을 기본으로 한다.
- 계산 기준, 포함 계좌, 기준 시각과 데이터 신선도를 설명할 수 있다.

### R9. 주·월·연 보고서

- 사용자는 주, 월, 연 기간을 전환해 수입, 지출, 순흐름, 예산 대비, 주요 카테고리와
  계좌별 내역을 본다.
- 모든 기간은 하나의 기간 경계 규칙과 타임존 규칙을 공유한다.
- 현재 진행 중 기간과 완료된 기간을 구분한다.
- 비교 보고서는 동일한 집계 정의를 사용한다.

### R10. 신선도와 부분 실패

- 계좌와 은행별 마지막 성공 동기화 시각을 표시한다.
- 한 은행이 실패해도 다른 은행 데이터는 사용할 수 있다.
- 누락 계좌가 있는 합계와 보고서는 불완전 상태로 명시하고 완전한 총액처럼 표현하지
  않는다.
- 재시도, 재인증 또는 지원 요청으로 이동할 수 있다.

### R11. 샘플·개발·운영 환경 구분

- fixture와 샘플 데이터는 실제 금융 연결과 동일한 도메인 계약을 사용해 제품 흐름을
  검증한다.
- 샘플, 개발 연동, 심사 중, 운영 연결을 화면과 운영 설정에서 구분한다.
- 샘플 연결만 존재하는 빌드는 1차 제품 출시로 간주하지 않는다.

### R12. 계정과 개인정보 수명주기

- 사용자는 앱 계정의 로그인 상태와 연결된 금융기관을 관리한다.
- 앱에서 계정 삭제를 시작할 수 있고, Google Play 요구에 맞는 외부 삭제 요청 경로도
  제공한다.
- 법적 보존 의무가 없는 개인·금융 연결 데이터는 삭제하고, 보존 대상은 범위와 기간을
  고지한다.
- 연결 해제, 은행 데이터 삭제와 앱 계정 삭제는 서로 다른 영향과 진행 상태를 가진
  별도 행위로 제공한다.
- 민감 토큰과 금융 데이터는 기기와 서버에서 최소 권한으로 보호한다.

### R13. 스토어·규제 준비

- 개인정보처리방침을 앱 내부와 스토어 메타데이터에서 접근 가능하게 한다.
- Apple 개인정보 항목, Google Play Data Safety와 금융 기능 선언을 실제 데이터 흐름과
  일치시킨다.
- 금융결제원 이용기관 승인·계약과 필요한 규제 판단을 운영 출시 전에 완료한다.
- 법률 자문 결과가 제품 데이터 흐름 변경을 요구하면 출시 범위를 재검토한다.

### R14. 개인정보 안전 관측성

- 장애율, 동기화 지연, connector 상태, 앱 충돌을 운영자가 확인할 수 있다.
- 로그, 분석, crash report에 계좌번호, 거래 원문, 잔액, 토큰과 직접 식별정보를 남기지
  않는다.
- 운영자 접근은 감사 가능하고 최소 권한을 따른다.

## 사용자 흐름

### F1. 가입과 첫 은행 연결

1. 사용자는 제품의 읽기 전용 약속과 개인정보 안내를 확인한다.
2. 앱 계정으로 로그인한다.
3. 은행을 선택하고 공식 인증·동의 흐름을 완료한다.
4. 연결 계좌와 조회 범위를 확인한다.
5. 첫 동기화 뒤 계좌별 마지막 성공 시각과 데이터 범위를 본다.

### F2. 동기화와 정합성 처리

1. connector가 잔액·거래를 조회한다.
2. 시스템은 원본을 멱등하게 저장한다.
3. 정규화가 자기계좌 이체, 중복 후보, 환불·취소와 거래처를 처리한다.
4. 검토가 필요한 항목을 사용자에게 표시한다.
5. 사용자 교정은 별도 이력으로 저장되어 다시 집계된다.

### F3. 예산 홈

1. 사용자는 계좌별 월 예산을 설정한다.
2. 앱은 은행 잔액과 남은 예산을 분리해 계산한다.
3. safe-to-spend, 예산 소진률, 지난달 동일 경과기간 대비를 본다.
4. 누락 계좌나 오래된 데이터가 있으면 지표와 함께 경고를 본다.

### F4. 기간 보고서

1. 사용자는 주·월·연 중 하나를 선택한다.
2. 앱은 동일한 기간·분류 규칙으로 수입, 지출, 순흐름과 예산 대비를 집계한다.
3. 계좌와 카테고리로 상세 내역을 추적한다.
4. 사용자가 분류를 교정하면 관련 보고서가 일관되게 갱신된다.

### F5. 장애·만료와 재인증

1. 특정 은행 연결이 지연, 만료 또는 장애 상태가 된다.
2. 앱은 영향받는 계좌와 마지막 성공 시각을 표시한다.
3. 나머지 계좌는 사용할 수 있지만 전체 합계는 불완전으로 표시한다.
4. 사용자는 재시도 또는 재인증 후 정상 상태를 확인한다.

### F6. 연결 해제, 은행 데이터 삭제와 계정 삭제

1. 사용자는 은행 연결 해제, 해당 은행 데이터 삭제 또는 앱 계정 전체 삭제를 선택한다.
2. 앱은 영향, 보존 대상과 삭제 범위를 설명한다.
3. 연결 해제는 토큰과 미래 동기화만 중단하고, 삭제 요청은 정책에 따라 원본·파생 데이터를 제거한다.
4. 사용자는 요청 접수와 완료 상태를 확인할 수 있다.

## 수용 기준

- **AC1 — 세 은행 공개 연결:** 출시 빌드에서 별도 사용자 계정으로 국민은행,
  카카오뱅크, 토스뱅크의 운영 계좌를 각각 연결하고 잔액·거래 조회와 해제를 검증한다.
- **AC2 — 읽기 전용 증명:** 모바일, 서버, connector 설정과 승인 범위 어디에도
  송금·결제 capability가 없으며, 보안 검토에서 돈을 움직이는 호출 경로가 발견되지
  않는다.
- **AC3 — 원장 정합성:** 자기 계좌 이체, 중복 거래, 카드대금 후보, 환불·취소와
  재동기화 fixture 검증에서 수입·지출·순흐름이 기대값과 일치한다. 카드대금 후보는
  사용자 제외 전 지출로 유지되고 제외·복원이 재현된다.
- **AC4 — 가계부와 가역성:** 카테고리와 메모를 수정하고 되돌렸을 때 원본은 보존되고
  모든 기간 집계가 같은 결과로 재계산된다.
- **AC5 — 예산 지표:** 월 중간 날짜에서 safe-to-spend, 소진률과 지난달 동일
  경과기간 비교가 정의된 기준 시각과 포함 계좌로 재현 가능하다.
- **AC6 — 기간 일관성:** 주·월·연 경계, 한국 시간대, 진행·완료 기간 자동 검증이
  통과하고 상세 내역 합계가 요약과 일치한다.
- **AC7 — 부분 실패:** 한 은행 응답을 중단하거나 토큰을 만료시켰을 때 다른 은행은
  계속 동작하고, 영향받는 합계에 누락 계좌와 마지막 성공 시각이 표시된다.
- **AC8 — 삭제와 철회:** 연결 해제 시 토큰과 미래 동기화만 중단되고 과거 이력은
  유지된다. 은행 데이터 삭제와 계정 삭제를 앱·웹 요청 경로에서 시작해 정책에 따른
  삭제·보존 결과를 각각 검증한다.
- **AC9 — 개인정보 안전:** 로그·분석·충돌 보고 샘플에 계좌번호, 거래 원문, 잔액,
  토큰과 직접 식별정보가 없다.
- **AC10 — 스토어 공개:** 동일 제품 계약의 iOS와 Android 출시 빌드가 각각 App
  Store와 Google Play에서 한국 사용자에게 공개되고 데이터 선언이 실제 동작과
  일치한다.

## 비기능 요구사항

- 보안: 금융 토큰과 개인정보에 최소 권한, 암호화, 회전·폐기와 감사 가능한 접근을
  적용한다.
- 정확성: 금액 계산은 부동소수점에 의존하지 않고 멱등 동기화와 재현 가능한 집계를
  제공한다.
- 회복성: 은행별 장애를 격리하고 재시도와 재인증을 구분한다.
- 성능: 일반적인 개인 계좌 수와 1년 거래량에서 주요 홈·보고서가 체감상 즉시 열리도록
  목표를 세우고 출시 전 측정한다.
- 접근성: iOS/Android 접근성 글꼴, 스크린리더, 색상 외 상태 표현을 검증한다.
- 시간: 모든 저장 시각과 보고서 기준 시간대가 명시되고 한국 달력 경계를 일관되게
  사용한다.
- 운영성: 금융 원문 없는 상태 지표와 추적 ID로 장애를 진단할 수 있다.
- 호환성: 지원 OS 범위는 계획 단계에서 Flutter·스토어 최신 지원 정책과 테스트 장비를
  기준으로 확정한다.

## 제품 가정과 승인된 결정

1. 첫 공개 시장은 대한민국이며 언어는 한국어, 통화는 원화다.
2. 1차 제품은 한 사람의 개인 명의 입출금 계좌만 다룬다.
3. 국민은행, 카카오뱅크, 토스뱅크 세 곳 모두가 출시 필수 조건이다.
4. 공식 오픈뱅킹 운영 승인과 규제 판단은 일정에 영향을 주는 외부 hard gate다.
5. 은행 OAuth 비밀과 동기화를 안전하게 처리하기 위해 모바일 앱과 별도의 서버
   구성요소가 필요하다.
6. 앱 계정을 제공하며 계정 삭제 흐름을 제품 범위에 포함한다.
7. 기존 ReplaceMe 저장소는 건드리지 않는다. 현재 `projects/sre-ai-lab` 저장소는
   별도 이력 보존·복구 ref 생성 없이 `projects/our-moneyflow` 제품 저장소로 직접
   전환한다. Git history rewrite나 강제 삭제는 수행하지 않는다.
8. 공식 승인 완료 전에는 출시일을 확정하지 않고 내부 배포와 공개 출시를 구분한다.
9. 금융 연결 경로는 KFTC 직접 연동과 운영 승인을 받은 공식 제공자를 비교한 뒤 선택한다.
10. 보안·삭제·신뢰 상태 UX·접근성과 출시 성능·beta 노출 기준은 선택 기능이 아니라
    공개 activation gate다.

P1~P7 기본안은 2026-07-21 사용자가 모두 승인했다. 후속 `ce-plan`은 이 계약을 9개 최소
배포 단위와 의사결정 게이트로 반영한다.
