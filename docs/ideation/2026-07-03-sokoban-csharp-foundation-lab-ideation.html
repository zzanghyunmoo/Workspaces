Grounding Context

이 아이디에이션은 구현 계획이 아니라 방향 후보를 고르는 문서다. 기준은 “기초 문법을 배우는 동안 지루하지 않고, 프로젝트 코드는 품질 습관을 같이 만들며, 6개월 뒤 확장 가능한 학습 포트폴리오가 되는가”다.

Repo context

  • README.md:3은 매일 1시간 C#/.NET 기본기 학습 repo라고 정의한다.
  • README.md:21-23은 긴 강의보다 작은 예제와 짧은 노트를 우선한다고 말한다.
  • README.md:27-36은 Sokoban Lab과 브랜치별 구현 원칙을 이미 잡아두었다.
  • README.md:49-87은 자료구조/알고리즘, 비동기, 네트워크, DB, 테스트 확장을 명시한다.

Sokoban context

  • project-kata-roadmap.md:7-13은 Sokoban이 턴 기반 격자라 TDD/DDD 학습에 적합하다고 본다.
  • project-kata-roadmap.md:26-41은 배열, 좌표계, 이동 규칙, 충돌 판정, solver, 순수 로직 테스트를 학습 소재로 든다.
  • project-kata-roadmap.md:49-67은 문자열 맵에서 시작해 value object, HashSet, Dictionary, Stack, Queue로 확장하는 흐름을 제안한다.

External baseline

  • Microsoft Learn: C# tour/tutorials are the canonical syntax baseline for variables, control flow, classes, collections, and async.
  • Microsoft Learn: object-oriented programming docs cover classes, objects, inheritance, polymorphism, encapsulation, and type choice.
  • Microsoft Learn: .NET testing guidance covers dotnet test and xUnit-based unit testing.
Key constraint: 작은 문법 예제는 탐색적으로 배워도 된다. 하지만 Sokoban에 들어가는 “프로젝트 코드”는 테스트를 먼저 작성하는 TDD 기본값을 가져야 한다.

Topic Axes

A1.학습 루프

공식 문서 → 간단 예제 → Sokoban 적용 → 회고가 매 챕터에서 반복되는가.

A2.코드 진화

절차지향으로 시작해 불편함을 체감한 뒤 객체지향/DDD-lite로 넘어가는가.

A3.TDD 경계

예제와 프로젝트 코드를 구분하고, 게임 규칙은 executable spec으로 고정하는가.

A4.브랜치 구조

base/ch01, datastructure/ch01처럼 확장되되 branch explosion을 막는가.

A5.확장 활주로

자료구조/알고리즘, 네트워크, DB, 디자인 패턴이 억지 과제가 아니라 Sokoban 필요에서 등장하는가.

I1.Three-Step Chapter Contract

Confidence 92% Complexity Low Axis A1
Description모든 챕터를 MS 공식 문서로 개념 확인 → 아주 작은 콘솔/함수 예제 → Sokoban 규칙에 TDD로 적용 → 5분 회고로 고정한다. 이 계약이 있으면 “기초 문법”과 “프로젝트 적용”이 분리되지 않고, 하루 1시간에도 반복 가능한 단위가 된다.
Basisdirect: README.md:21-22는 매일 1시간과 작은 예제/짧은 노트 우선 원칙을 둔다. external: Microsoft Learn의 C# tour/tutorials를 기본 문법 출처로 삼을 수 있다.
Rationale사용자는 만들면서 배우는 성향이지만, 곧장 프로젝트로 들어가면 문법 구멍이 생긴다. 이 구조는 공식 문서의 신뢰도와 프로젝트 학습의 재미를 동시에 잡는다.
Downsides챕터 작성 시 매번 공식 문서 링크와 작은 예제를 붙여야 해서 문서 작성량이 늘어난다.

I2.Two-Lane Code Policy

Confidence 90% Complexity Low Axis A3
Descriptionrepo를 “예제 lane”과 “Sokoban project lane”으로 사고상 분리한다. 문법 예제는 빠르게 실험해도 되지만, Sokoban 규칙/파서/상태 전이는 TDD-first로만 진행한다.
Basisdirect: 사용자 조건은 “간단 예제를 배우고 프로젝트에 적용”과 “예제가 아닌 프로덕션 코드는 TDD 기본”을 동시에 요구한다. project-kata-roadmap.md:111-115도 domain logic 테스트를 강조한다.
RationaleTDD를 모든 문법 예제에 강제하면 초반 학습 속도가 느려진다. 반대로 프로젝트 코드까지 테스트 없이 가면 품질 습관이 생기지 않는다. 두 lane 정책은 둘 사이의 긴장을 명확히 해소한다.
Downsides“이 코드는 예제인가 프로젝트인가” 경계를 문서에 명확히 적어야 한다. 경계가 흐리면 규칙이 유명무실해진다.

I3.Procedural-to-OOP Migration Ladder

Confidence 88% Complexity Medium Axis A2
Description처음부터 Board, GameSession, DomainService를 만들지 않는다. 먼저 char[,], 좌표 변수, 절차형 함수로 불편함을 일부러 경험한 뒤, 중복·상태 꼬임·테스트 난이도를 근거로 객체를 도입한다.
Basisdirect: project-kata-roadmap.md:49-67은 문자열 맵 → 2차원 배열 → value object → HashSet/Dictionary/Stack/Queue → domain model 흐름을 이미 제안한다.
Rationale초보자가 OOP를 “외워야 할 구조”가 아니라 “절차형 코드가 버티지 못해서 필요한 도구”로 이해하게 된다. 이것이 나중에 DDD를 받아들이는 가장 자연스러운 진입점이다.
Downsides초반 브랜치의 코드는 의도적으로 덜 예쁘다. 문서에 “이 지저분함은 다음 챕터의 재료”라고 명확히 써야 한다.

I4.Executable Rulebook

Confidence 86% Complexity Medium Axis A3
DescriptionSokoban 규칙을 “설명 문서”가 아니라 테스트 목록으로 만든다. 예: 벽으로 이동 불가, 상자 뒤가 빈 칸이면 밀 수 있음, 상자 뒤가 벽이면 이동 불가, 모든 상자가 goal 위에 있으면 clear.
Basisdirect: project-kata-roadmap.md:117-124는 바로 이런 테스트 문장 후보를 제시한다. external: Microsoft Learn의 .NET unit testing/xUnit 문서를 공식 테스트 기준으로 삼을 수 있다.
Rationale이 방식은 TDD를 이론으로 가르치지 않는다. “게임 규칙을 먼저 문장으로 쓰고, 그 문장이 테스트가 되고, 테스트가 구현을 이끈다”는 몸의 감각을 만든다.
Downsides초반에는 콘솔 출력보다 순수 로직을 먼저 분리해야 해서, 눈에 보이는 게임 화면은 조금 늦게 나온다.

Downsides브랜치 간 diff가 학습 자료가 되려면 각 브랜치 README/notes 작성 습관이 필요하다.

I6.Concept Pressure Ladder

Confidence 82% Complexity Medium Axis A5
Description자료구조/알고리즘/네트워크/DB/패턴을 독립 단원으로 던지지 말고, Sokoban이 “압력을 주는 순간”에 도입한다. 충돌 판정이 느려지면 HashSet, undo가 필요하면 Stack, solver가 필요하면 Queue/BFS, 레벨 공유가 필요하면 HTTP, 진행 기록이 필요하면 DB, undo 명령이 반복되면 Command pattern으로 간다.
Basisdirect: README.md:49-79는 DS/A, async/file, network/API, DB를 6개월 확장 축으로 둔다. project-kata-roadmap.md:142-187은 자료구조와 알고리즘이 Sokoban 문제에서 자연스럽게 등장하는 예를 든다.
Rationale개념을 외우는 커리큘럼이 아니라 “지금 구조로는 불편해서 더 좋은 도구를 도입한다”는 학습 흐름이 된다. 이 흐름은 만들면서 배우는 성향과 잘 맞는다.
Downsides초반 문서에서 미래 확장을 너무 많이 보여주면 산만해질 수 있다. 본편과 extension track의 경계를 유지해야 한다.

I7.DDD-Lite Language Refactor

Confidence 78% Complexity Medium Axis A2
DescriptionDDD는 초반 아키텍처가 아니라 “이름 붙이기 리팩터링”으로 도입한다. int row, int col이 반복되면 Position, 문자 비교가 복잡해지면 Cell/Tile, 이동 결과가 애매해지면 MoveResult, 상태 일관성이 중요해지면 GameState 또는 GameSession으로 이동한다.
Basisdirect: project-kata-roadmap.md:189-227은 DDD를 과하게 도입하지 말고 도메인 언어를 코드에 반영하라고 정리한다. external: Microsoft Learn OOP docs는 class/object/type choice를 공식 배경으로 제공한다.
RationaleDDD를 “Repository/Service/Factory 폴더 구조”가 아니라 “도메인 언어를 코드로 옮기는 훈련”으로 배우게 한다. 초급자에게 부담이 적고, 나중에 실제 비즈니스 도메인으로 넘어갈 때도 전이가 쉽다.
DownsidesDDD 용어를 너무 빨리 많이 쓰면 본래 목표인 C# 기초가 흐려진다. Value Object와 Entity 정도로 시작하고, Repository는 DB 확장 시점까지 늦추는 편이 안전하다.

Rejection Summary

# Idea Reason Rejected
R1. Full DDD / Clean Architecture from chapter 1 절차지향 → 객체지향 학습 요구와 충돌한다. 초반 인지부하가 너무 높다.
R2. Build UI/Blazor first 눈에 보이는 재미는 있지만, domain logic 테스트와 기본 문법 학습을 흐릴 가능성이 크다.
R3. Require TDD for every tiny syntax example 프로젝트 코드 TDD 원칙은 중요하지만, 모든 문법 실험에 강제하면 학습 속도를 떨어뜨린다.
R4. One giant implementation branch 브랜치별 학습 스냅샷과 diff 기반 회고라는 사용자 요구를 살리지 못한다.
R5. Separate unrelated mini-project per chapter 각 개념은 배우기 쉬워질 수 있지만, 하나의 프로젝트가 계속 진화하는 compounding 효과가 사라진다.
R6. Tetris as the main foundation project 좋은 후보지만 실시간 루프와 렌더링 부담이 Sokoban보다 빨리 커진다.
R7. Mini Log Query Engine as the main foundation project SRE 방향과는 잘 맞지만, 초반 기초 프로그래밍 학습에서 시각적/게임적 피드백이 약하다.
R8. Start DB/network in month 1 확장성은 좋지만 C# 기본 문법과 절차형 사고를 충분히 익히기 전에는 범위 과잉이다.
R9. Branch per tiny concept 학습 단위가 너무 잘게 쪼개져 관리 비용이 학습 효과를 앞지를 수 있다.
R10. SRE AI Lab과 foundation repo를 다시 합치기 이미 제품/콘텐츠 작업과 기초 학습을 분리한 결정과 충돌한다.
