Evidence before ideas

Grounding Context

현재 저장소

projects/sre-ai-lab은 README와 Alert Rule Clinic 문서만 있는 별도 Git 저장소다. Flutter 코드·manifest·테스트·remote가 없다.

정체성 충돌

SRE AI Lab, 별도 ReplaceMe submodule/DevAutomation 문서, 새 our-moneyflow 의도가 동시에 존재한다. root gitlink에는 .gitmodules 매핑도 없다.

금융 연결 현실

KB국민·카카오뱅크·토스뱅크는 오픈뱅킹 참여기관이며 조회 API·OAuth 기반이 있다. 운영 이용은 승인·계약 등 외부 관문이 있고 MyData 해당성은 별도 판정이 필요하다.

제품 기회

경쟁 앱은 이미 자산 연결·가계부·소비 분석을 제공한다. 계좌 = 예산 봉투와 행동 중심 safe-to-spend는 검증할 제품 가설이다.

정합성 위험

자기계좌 이체, 카드대금 중복, 환불·취소, 거래처 정규화, 분류 수정, 잔액 구분, 기간 경계와 데이터 신선도가 보고서 신뢰를 좌우한다.

재사용할 유산

기존 기술 구조는 거의 없지만 1~1.5시간 단위로 작게 검증하는 작업 리듬은 은행·원장·보고 vertical slice에 재사용할 수 있다.

판단 유보. 이 문서는 법률 의견이나 구현 계획이 아니다. 오픈뱅킹 운영 절차와 MyData 적용 여부는 서비스 흐름·저장 데이터 명세를 기준으로 금융결제원과 자격 있는 전문가에게 재확인해야 한다.

1.

제품 검증 트랙

중심 축 · 예산·기간 비교·주월연 보고서 흡수 · 기존 1 + 2 + 5 Confidence · 88% Complexity · High
Description
하나의 읽기 전용 금융 피드 계약 뒤에 샘플·CSV와 공식 connector를 둔다. 샘플 데이터로 계좌 = 예산 봉투 홈, 은행 잔액과 예산 잔액의 분리, 이번 주 안전 사용액, 지난달 동일 경과기간 비교, 주·월·연 결산을 먼저 검증한다. 동시에 공식 오픈뱅킹 연동과 규제 판정을 진행하며 화면은 샘플, 개발 연동, 심사 중, 운영 연결을 명확히 구분한다.
Basis
external + product hypothesis: 금융결제원 조회 API는 승인·계약이라는 외부 관문이 있다. 경쟁 앱이 이미 종합 자산·소비 분석을 제공하므로, 우리 가설은 “목적별 계좌에서 지금 얼마를 더 써도 되는가”에 집중한다.
Rationale
제도 관문 때문에 사용자 가치 학습이 멈추지 않는다. 같은 기간 엔진을 홈과 주·월·연 보고가 공유하므로 비교 숫자가 일치하고, 정식 connector가 도착해도 제품 코어를 버리지 않는다.
Downsides
fixture가 실제 금융 데이터의 지연·오류·은행별 차이를 과소대표할 수 있다. 계좌 목적과 실제 소비 범위가 맞지 않는 사용자도 있으며, 기간 결산은 회계 도구처럼 무겁게 느껴질 수 있다.
2.

금융 신뢰 기반

중심 축 · 금융 데이터 신뢰·프라이버시·연결 상태 흡수 · 기존 3 + 4 + 6 Confidence · 91% Complexity · High
Description
은행 원본 거래를 불변으로 두고 원본 → 정규화된 경제적 사건 → 사용자 교정·분류 → 집계로 분리한다. connector에는 조회 capability만 두고 동의 범위·철회·보존·삭제를 확인할 수 있게 한다. 은행별 연결을 정상, 지연, 만료, 장애, 재인증 필요, 해제로 독립 관리하고 모든 잔액·예산·보고서에 마지막 성공 시각과 누락 계좌를 붙인다.
Basis
external + reasoned: OAuth 기반 조회는 동의·토큰 수명과 기관별 장애 경계를 가진다. 여러 은행 레코드를 그대로 수입·지출로 세면 이체·카드대금·환불이 중복되므로 원본과 해석을 분리해야 재계산과 설명이 가능하다.
Rationale
“돈을 움직일 수 없다”, “왜 이 숫자인가”, “얼마나 최신인가”를 하나의 신뢰 계약으로 묶는다. 한 은행의 문제를 전체 앱 실패로 번지지 않게 하면서 불완전한 합계를 완전한 숫자처럼 보이지 않는다.
Downsides
원장 모델, 토큰·보존 정책, 상태 표현과 정합성 테스트 부담이 크다. 연결 상태가 많아지면 홈이 복잡해질 수 있으며 실제 저장 방식은 제휴·규제 판정 뒤에 확정해야 한다.
3.

프로젝트 전환

중심 축 · 프로젝트 정체성·저장소 전환 유지 · 기존 7 Confidence · 96% Complexity · Low
Description
Flutter 작업 전에 canonical repo·remote·기본 브랜치·workspace submodule 경로·README·Notion·Linear를 한 전환 계약으로 고정한다. SRE AI Lab과 ReplaceMe는 새 제품의 별칭이 아니라 보존·종료·이관 상태가 있는 이전 정체성으로 처리한다.
Basis
direct: 현재 projects/sre-ai-lab README는 SRE AI Lab/C# 제품을 설명하고 remote가 없으며, root gitlink에는 .gitmodules 매핑이 없다. ReplaceMe는 별도 submodule과 제품 문서를 가진다.
Rationale
첫 Flutter 커밋부터 코드·문서·티켓이 세 정체성으로 갈라지는 것을 막는다.
Downsides
제품 가치 검증과 직접 관련 없는 선행 정리처럼 보일 수 있다. 최종 canonical 저장소 선택은 사용자의 명시 결정이 필요하다.

Rejection Summary

#IdeaReason Rejected or Merged
1두 트랙 MVP + 계좌 예산 조종석 + 기간 대수와 결산 사슬제품 검증 트랙으로 병합. 제품 학습, 핵심 행동 모델, 기간 계산을 하나의 검증 방향으로 묶었다.
2설명 가능한 경제적 사건 원장 + 읽기 전용 계약 + 연결 circuit breaker금융 신뢰 기반으로 병합. 정확성, 권한 경계, 데이터 신선도는 분리할 수 없는 하나의 신뢰 계약이다.
3금융 데이터 영수증과 프라이버시 종료금융 신뢰 기반에 병합. 기존 direct 근거는 secret 취급만 지지하고 전체 보존 정책까지 지지하지 않았다.
4계좌별 생활 일지제품 검증 트랙에 병합. 별도 탐색 구조보다 계좌 = 예산 봉투 제품 모델의 일부가 더 강하다.
5은행 거래가 아닌 경제적 사건 원장금융 신뢰 기반의 설명 가능한 원장에 병합된 직접 중복이다.
63문장 주간 보고서제품 검증 트랙의 전달 형식으로 병합. 독립적인 행동 변화 근거가 약했다.
7주→월→연 결산 사슬제품 검증 트랙의 기간 엔진에 병합해 계산 규칙과 사용자 마감을 한 방향으로 유지했다.
8남은 약속과 읽기 전용 시뮬레이션기본 원장 신뢰 전에 예측 정확성과 범위를 늘려 MVP 후보에서 제외했다.
9모든 숫자의 데이터 계보 칩금융 신뢰 기반의 공통 UI 메커니즘으로 병합했다.
10한 계좌로 끝나는 점진적 온보딩제품 검증 트랙의 범위 축소 전술로 병합. 이탈 감소 효과는 아직 입증되지 않았다.
