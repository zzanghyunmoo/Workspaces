north star

6개월 뒤 갖고 있어야 하는 것

6개월 뒤에 “뭔가 많이 공부했다”가 아니라, 아래 네 가지가 남아 있어야 한다.

  • 작동하는 작은 제품
    Prometheus/Mimir alert rule을 분석하는 Alert Rule Clinic MVP.
  • 공개 포트폴리오
    .NET, AI agent, SRE 도메인이 함께 보이는 GitHub repo와 demo.
  • 반복 가능한 콘텐츠 포맷
    블로그, 유튜브, 체크리스트가 한 기능에서 같이 나오는 구조.
  • 초기 수요 신호
    댓글, DM, 뉴스레터/폼 신청, 파일 다운로드, GitHub star 같은 작은 반응.
이 로드맵의 핵심은 제품 완성이 아니라 제품·학습·콘텐츠가 한 번에 굴러가는지 검증하는 것이다. 6개월 안에 큰 플랫폼을 만들려고 하면 오히려 방향을 잃기 쉽다.

operating rules

로드맵 원칙

  • 하나의 중심 제품만 둔다.
    처음 6개월의 중심은 Alert Rule Clinic이다.
  • .NET은 보여주기용이 아니라 실제 제품의 control plane으로 쓴다.
    ASP.NET Core, CLI, agent orchestration을 실제로 만진다.
  • 기본기는 따로 챙기되, 제품과 분리하지 않는다.
    C#, 자료구조/알고리즘, 네트워크, DB는 초반에 정리하고 이후 Alert Rule Clinic 안에서 계속 확인한다.
  • 기능 하나를 만들면 콘텐츠도 같이 만든다.
    코드, 글, 영상, 체크리스트가 한 묶음으로 나온다.
  • money-flow와 로드맵 경계를 분리한다.
    경제 블로그는 별도 프로젝트에서 운영하고, 여기서는 SRE AI 제품과 개발 콘텐츠에 집중한다.
  • 게임 개발은 timebox한다.
    Unity 실험은 SRE 개념 설명용으로만 작게 가져간다.
  • 매달 공개 산출물을 남긴다.
    비공개 공부만 하면 브랜드와 수요 검증이 쌓이지 않는다.

foundation runway

.NET 기본기 레인

닷넷을 잘 모르는 상태라면 기본기는 따로 정리하고 가는 게 맞다. 다만 제품을 미뤄두고 공부만 오래 하는 방식은 피한다. 처음 2~4주는 기본기 비중을 높이고, 이후에는 Alert Rule Clinic을 만들면서 필요한 부분을 계속 보강하는 구조가 좋다.

  • C# / .NET 기본
    type system, nullable, record, pattern matching, LINQ, async/await, collection, exception, logging, configuration, dependency injection.
  • 자료구조 / 알고리즘
    코딩테스트식으로 깊게 파기보다 List, Dictionary, Set, Queue, Stack, Tree, Graph, 정렬, 탐색, Big-O를 제품 코드 관점에서 정리한다.
  • 네트워크 / API
    HTTP, REST, gRPC 개념, timeout, retry, cancellation, idempotency, serialization, auth 기본을 ASP.NET Core와 연결해서 본다.
  • DB / 저장소
    PostgreSQL 기준 SQL, index, transaction, isolation, migration, EF Core와 Dapper의 차이를 가볍게 잡는다.
추천 리듬: 기본기는 제품 작업과 분리해서 매일 1시간씩 가져간다. 제품과 콘텐츠 작업은 주 2~3회, 회당 1~1.5시간 블록으로 잡는다. 기본기는 “강의 완강”보다 작은 예제와 제품 코드로 확인하는 쪽이 낫다.

weekly cadence

주간 실행 리듬

이 로드맵은 시간을 많이 쓰는 전업 프로젝트가 아니라, 꾸준히 쌓는 사이드 프로젝트에 가깝다. 그래서 매일 긴 시간을 잡기보다, 기본기와 제품 작업을 분리해서 리듬을 유지하는 편이 낫다.

  • 매일 1시간 · .NET 기본기
    C# 문법, LINQ, async/await, 자료구조, 네트워크, DB를 짧은 예제로 정리한다. 이 시간에는 제품 욕심을 내지 않는다.
  • 주 2~3회 · 제품 작업
    회당 1~1.5시간만 Alert Rule Clinic 구현에 쓴다. 작은 issue 하나를 끝내는 단위로 잡는다.
  • 주 1회 · 콘텐츠 정리
    그 주에 배운 것과 만든 것을 블로그 초안, 영상 메모, 체크리스트 중 하나로 남긴다. 완성보다 기록을 우선한다.
  • 월 1회 · 공개 산출물
    README, demo, 글, 영상, 체크리스트 중 하나는 반드시 외부에 공개한다.
중요한 운영 규칙: 기본기 1시간은 매일 고정하고, 제품 작업은 욕심내서 매일 하지 않는다. 주 2~3회만 해도 6개월이면 충분히 누적된다.
Q3

2026 Jul → Sep

검증과 첫 자산 만들기

목표: MVP 감 잡기 산출: repo + 글 + 영상

Q3의 목적은 완성품이 아니다. Alert Rule Clinic이 정말 좋은 첫 wedge인지 확인하고, .NET으로 이 문제를 풀 수 있다는 감을 잡고, 동시에 콘텐츠 포맷을 테스트하는 분기다.

7월 · 방향 고정과 샘플 수집

  • 매일 1시간씩 C#/.NET 기본 문법과 project 구조를 정리한다.
  • 주 2~3회, 회당 1~1.5시간으로 Alert Rule Clinic repo를 만든다.
  • 좋은 alert rule / 나쁜 alert rule 기준을 1차로 적는다.
  • 샘플 Prometheus rule 10개를 모은다.
  • ASP.NET Core 또는 .NET CLI skeleton을 만든다.
  • 블로그 1편: 왜 alert rule review가 필요한가.

8월 · 첫 MVP와 리뷰 리포트

  • 제품 작업일에는 YAML 입력을 받아 rule을 파싱한다.
  • 매일 기본기 시간에는 자료구조/알고리즘을 rule model, collection, 탐색 로직과 연결해 정리한다.
  • 휴리스틱 체크 5~7개를 만든다.
  • LLM 리뷰 프롬프트를 붙인다.
  • markdown report를 생성한다.
  • 유튜브 1편: AI로 Mimir alert rule 리뷰해보기.

9월 · 공개 데모와 피드백

  • 샘플 rule corpus를 20~30개로 늘린다.
  • GitHub README와 demo gif를 정리한다.
  • 간단한 웹 UI 또는 CLI 사용성을 다듬는다.
  • 피드백 폼을 열고 댓글/DM/폼 반응을 모은다.
  • 체크리스트 PDF 또는 markdown 자료를 배포한다.
Q3 성공 기준: 완벽한 제품이 아니라 “이걸 계속 만들 만한가?”를 판단할 신호가 있어야 한다. 예: repo 공개, 데모 1개, 글 2~3개, 영상 1~2개, 피드백 5개 이상.
Q4

2026 Oct → Dec

반복과 초기 수익화 준비

목표: 반복 가능한 채널 산출: mini course seed

Q4는 Q3의 데모를 한 번 더 밀어붙이는 분기다. 여기서 중요한 건 기능을 크게 늘리는 것이 아니라, 제품·콘텐츠·강의 자료가 같은 주제로 반복 생산되는지 보는 것이다.

10월 · Demo Bank와 Runbook Card

  • synthetic incident scenario 5개를 만든다.
  • alert-rule smell catalog를 정리한다.
  • SRE Micro-Runbook card 포맷을 정한다.
  • 기존 kubectl request/limit 문서를 첫 card로 바꾼다.
  • 블로그 2편: alert smell / runbook card.

11월 · .NET Lab과 콘텐츠 루틴

  • .NET AI SRE Lab 문서 구조를 정리한다.
  • Alert Rule Clinic을 lab episode 1~3으로 쪼갠다.
  • money-flow와 겹치지 않도록 콘텐츠 경계를 정리한다.
  • 뉴스레터 또는 구독 폼을 만든다.
  • 유튜브 2편 또는 긴 영상 1편을 낸다.

12월 · 작은 패키지와 다음 선택

  • 무료 자료: Alert Rule Review Checklist v1을 배포한다.
  • 미니 강의 outline을 만든다.
  • Q3~Q4 반응을 숫자로 정리한다.
  • 계속할지, 좁힐지, 다른 wedge로 바꿀지 결정한다.
  • 2027 Q1 계획 후보를 2~3개로 압축한다.
Q4 성공 기준: 제품 기능보다 반복 가능성이 중요하다. 글/영상/자료가 꾸준히 나오고, 누군가 “이거 써보고 싶다” 또는 “이 주제로 더 알려달라”고 말하면 성공이다.

decision gates

분기 말 의사결정

로드맵은 계속 밀어붙이는 문서가 아니라, 분기 말에 방향을 고르는 장치여야 한다. 특히 6개월짜리 실험에서는 아래 질문에 솔직하게 답하는 게 중요하다.

Q3 말: Alert Rule Clinic을 계속 밀어도 되는가? repo, 글, 영상, 피드백을 보고 판단한다. 피드백이 거의 없더라도 내가 계속 만들고 싶은지, 콘텐츠가 자연스럽게 나오는지 확인한다.
Q4 말: 제품 중심으로 갈까, 교육 중심으로 갈까? 제품 사용 요청이 많으면 MVP를 키운다. 질문과 학습 수요가 더 많으면 mini course와 자료 쪽을 먼저 키운다. 둘 다 약하면 wedge를 바꾼다.
2027 Q1 후보 6개월 뒤에는 두 가지 중 하나를 우선 고르면 된다. ① Alert Rule Clinic을 제품으로 확장, ② .NET AI SRE Lab을 강의/콘텐츠 시리즈로 확장. 경제 블로그는 별도 money-flow 프로젝트의 로드맵에서 판단한다.

not now

6개월 동안 일부러 하지 않을 것

  • 처음부터 SaaS 결제, 멀티테넌시, 조직 관리 기능 만들기
  • Slack/Teams, PagerDuty, Grafana 등 모든 integration을 붙이기
  • money-flow 프로젝트와 이 로드맵을 섞어서 운영하기
  • Unity 게임을 별도 장기 프로젝트로 키우기
  • 콘텐츠 자동화 플랫폼을 먼저 다시 만들기

이것들은 나중에 할 수 있다. 하지만 첫 6개월에는 중심축이 흐려지지 않는 게 더 중요하다.
